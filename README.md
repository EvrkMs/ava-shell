# ava-shell

`ava-shell` это платформа для управления операционной работой компьютерного клуба, вдохновлённая такими продуктами, как `LANGAME`, но изначально задуманная как открытая система, а не как закрытая legacy-панель.

Проект появился потому, что существующие решения обычно закрывают задачи клуба только частично. Они помогают вести операционку, но слабы в интеграциях, слабы в public API и часто плохо отражают то, что реально произошло в рабочих процессах. Из-за этого клубам всё равно приходится писать собственные инструменты для контроля смен, проверки расхождений, учёта сейфа, аналитики и проверки отчётов менеджером.

## Суть проекта

`ava-shell` задумывается в двух слоях:

- готовый продукт для ежедневной работы клуба;
- платформа с public API, через которое доступны почти все важные действия.

Цель не только в том, чтобы дать администраторам и менеджерам ещё одну панель. Цель в том, чтобы дать клубу надёжное операционное ядро, которое можно расширять своими desktop-приложениями, аналитикой, CRM, автоматизациями и внутренними процессами.

## Зачем он нужен

У продуктов на этом рынке обычно есть одна или несколько проблем:

- интеграции ограничены, хрупкие или отсутствуют;
- public API неполный или нестабильный;
- аналитика заперта внутри готовых экранов вместо доступа к сырым данным;
- система навязывает один идеальный workflow вместо поддержки реальных сценариев;
- расхождения скрываются через обходные операции вместо того, чтобы фиксироваться напрямую.

На практике это приводит к таким ситуациям:

- чтобы оприходовать товар, сначала нужно создать искусственный заказ;
- закрытие смены не даёт нормально зафиксировать фактические значения;
- расхождения смешиваются с обычными расходами;
- проверка отчётов уходит во внешние самописные инструменты и ручной контроль.

`ava-shell` создаётся именно для того, чтобы не повторять такие продуктовые решения.

## Основной принцип

Система должна хранить не только то, что ожидала операционная программа, но и то, что реально произошло.

Это касается:

- фактических значений по смене;
- итогов по терминалу с подтверждением фото чека;
- налички, пересчитанной при закрытии смены;
- остатка в сейфе после того, как деньги убрали из кассы;
- склада и приёмки по факту, а не только по идеальному сценарию закупки;
- проверки менеджером и разбора расхождений.

Продукт должен делать реальность видимой, а не заставлять сотрудников имитировать “правильный” процесс.

## Направление платформы

По духу `ava-shell` ближе к `МойСклад`, чем к закрытым клубным системам:

- public API является частью продукта с самого начала;
- почти каждое важное действие должно быть доступно не только через UI;
- внешние системы должны уметь читать сырые данные и выполнять операционные действия;
- клуб должен иметь возможность писать свой локальный софт поверх платформы;
- интеграции должны быть частью архитектуры, а не случайным дополнением.

Это значит, что `ava-shell` должен быть не только готовым приложением, но и основой для собственной инфраструктуры клуба.

## Видение продукта

Долгосрочная цель это полноценная открытая платформа для управления компьютерным клубом:

- отчёты по сменам и контроль расхождений;
- сверка кассы, терминала и сейфа;
- склад, приёмка и движения товаров;
- процессы сотрудников и менеджеров;
- аналитика на основе доступных сырых данных;
- интеграции с внешними системами;
- собственные desktop-инструменты и внутренние сервисы на тех же публичных контрактах.

Коротко:

`ava-shell` должен быть удобен как готовый продукт, но спроектирован как платформа.
