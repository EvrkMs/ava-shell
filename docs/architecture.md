# Архитектурные заметки

Этот документ нужен для обсуждения реализации `ava-shell`. В отличие от `README`, здесь фиксируются не продуктовые тезисы, а текущие технические решения, ограничения и направление развития.

Документ не является финальной спецификацией. Это рабочая база для архитектурных решений, которую можно уточнять по мере появления MVP.

## Текущий подход

На текущем этапе важнее не максимальная масштабируемость любой ценой, а предсказуемая разработка, понятный деплой и возможность спокойно развивать продукт в одиночку.

Из этого следуют стартовые принципы:

- сначала чистые границы модулей, потом выделение в отдельные сервисы;
- сначала `dev`-среда и удобный цикл разработки, потом полноценный `prod`;
- инфраструктура должна быть простой в сопровождении одним человеком;
- все решения должны оставлять путь к росту, но не тащить лишнюю сложность заранее.

## Архитектура приложения

Базовый ориентир на старте это не набор микросервисов, а модульная архитектура с возможностью последующего разделения.

Практически это означает:

- один репозиторий;
- разделение по доменам и сценариям;
- явные контракты между частями системы;
- возможность вынести модуль в отдельный сервис без переписывания предметной модели.

Микросервисная структура может появиться позже, если для этого возникнут реальные причины:

- независимое масштабирование;
- разные требования к отказоустойчивости;
- разные типы нагрузки;
- необходимость отдельно релизить части платформы.

## Контуры доступа

Система проектируется минимум в трёх контурах:

### Public

Контур для пользовательского и внешнего доступа из интернета.

Сюда относятся:

- admin panel;
- web-интерфейсы для менеджеров и владельцев;
- публичное API для интеграций;
- SSO и пользовательская аутентификация.

Ожидания по безопасности:

- обычный `TLS`;
- пользовательская аутентификация;
- аудит действий;
- ограничения по rate limit;
- защита публичной поверхности.

### Private

Контур для доверенных машин заказчика и клубной инфраструктуры.

Сюда относятся:

- агент на ПК клуба;
- служебные machine-to-machine взаимодействия;
- синхронизация с локальным софтом;
- лицензирование и идентификация инсталляций.

Ожидания по безопасности:

- `mTLS`;
- идентификация узла, инсталляции или клиента;
- отдельные endpoint'ы и политики доступа;
- отсутствие смешения с пользовательским веб-трафиком.

### Internal

Полностью закрытый контур платформы, недоступный даже заказчику.

Сюда относятся:

- внутренние сервисы платформы;
- служебные API;
- фоновые worker'ы;
- выпуск и отзыв сертификатов;
- служебные панели сопровождения;
- внутренние базы и очереди.

Ожидания по безопасности:

- недоступность из внешней сети;
- отдельные права доступа;
- отдельные секреты;
- отдельный операционный контроль.

## TLS, PKI и mTLS

В проекте ожидается использование собственного SSO и собственного `mTLS` для доверенных машин.

При этом важно разделять две задачи:

- сертификат как средство установления доверенного канала;
- лицензия как средство продуктового ограничения и идентификации заказчика.

Сертификаты не должны становиться единственным носителем всей бизнес-логики лицензирования. Лучше разделять:

- транспортную идентификацию через сертификат;
- продуктовые права и лимиты через модель лицензии.

### Dev

Для `dev` допустим собственный `CA` и собственная DNS-схема, если это действительно внутренний контур команды и инфраструктуры.

Подход:

- внутренний корневой `CA`;
- сертификаты для `dev`-доменов;
- установка доверенного корня на машины разработчиков и нужные серверы;
- использование этого контура для разработки собственного SSO и `mTLS`.

### Prod

Для `prod` предполагается обычная публичная схема:

- публичные домены;
- сертификаты от общедоверенного `CA`;
- отдельный внутренний контур для закрытых сервисов.

## Окружения

Пока у проекта нет `MVP`, основным рабочим окружением считается `dev`.

Отдельный `prod` до появления минимально жизнеспособного продукта не обязателен. Он добавит операционные расходы раньше, чем даст реальную пользу.

При этом уже сейчас стоит проектировать инфраструктуру так, чтобы `prod` можно было добавить без переделки всей схемы:

- конфигурация по окружениям разделена;
- секреты не зашиты в код;
- деплой работает через переменные окружения и артефакты;
- образы и релизы версионируются.

## Политика миграций БД

Для `dev` потеря части данных иногда допустима. Для `prod` такой подход неприемлем.

Поэтому миграции БД должны проектироваться не только как изменение схемы, но и как безопасный переход между версиями приложения без потери данных.

### Общий принцип

Для рабочих окружений используется подход `expand -> migrate -> contract`.

Это означает:

- сначала добавляется новая структура;
- потом приложение и данные переводятся на неё;
- старая структура удаляется только позже, когда она точно больше не используется.

### Правила

- нельзя удалять колонку или таблицу в той же миграции, где появилась её замена;
- удаление допустимо только после подтверждённого отказа от использования;
- если данные нужны, они должны быть заранее перенесены;
- если данные больше не нужны, это должно быть явным и осознанным решением;
- destructive changes не должны попадать в рабочее окружение без отдельной проверки.

### Переименование

Если колонка или таблица переименовывается, нужно выполнять именно переименование, а не схему вида `удалить старое и создать новое`.

Практически это означает:

- использовать явные операции переименования;
- не принимать автосгенерированные миграции вслепую;
- проверять, не заменил ли генератор переименование на `drop/add`.

### Перенос данных

Если данные перемещаются в другие колонки или таблицы, это должно оформляться как отдельный этап миграции.

Подход:

- добавить новую структуру;
- перенести данные;
- перевести приложение на новую модель;
- только после этого планировать удаление старой структуры.

### Удаление старой структуры

Удаление колонок и таблиц должно быть отложенным.

Базовое правило:

1. в первой миграции добавляется новая структура;
2. в следующей миграции или релизе выполняется переход кода и данных;
3. удаление старой структуры допускается только в одной из последующих миграций, когда она точно больше не используется.

Иными словами, колонка не удаляется сразу. Сначала она становится legacy, перестаёт использоваться и только потом может быть удалена.

### Разделение по окружениям

Для `dev` возможны более свободные миграции, если это ускоряет разработку.

Для рабочих окружений будущего `prod` действуют более строгие правила:

- ручная проверка каждой миграции;
- запрет на слепое применение destructive migration;
- резервное копирование перед изменениями схемы;
- отдельное внимание к переименованию и переносу данных.

## CI/CD

Текущий ориентир:

- `GitHub Actions` как основной CI/CD-инструмент;
- `GitHub Environments` для раздельных секретов и правил доступа;
- автоматический деплой в `dev` по коммитам;
- деплой в `prod` по релизу или тегу, когда появится отдельная prod-среда.

### Secrets

Секреты должны жить в `GitHub`, но не в репозитории:

- отдельный `Environment` для `dev`;
- отдельный `Environment` для `prod`;
- разные наборы секретов;
- отдельные approval-правила для `prod`.

### Runner

Для self-hosted runner на текущем этапе предпочтительнее установка прямо на хост, а не запуск внутри Docker.

Причины:

- проще настройка;
- проще диагностика;
- меньше проблем с доступом к Docker;
- меньше лишней инфраструктурной магии для одного разработчика.

Runner-хост не должен совпадать с будущим runtime-окружением `prod`, если есть возможность их разделить.

## Registry и доставка образов

На старте нет необходимости обязательно поднимать собственный registry.

Практичный путь:

- использовать внешний registry, например `GHCR`;
- собирать и публиковать образы из CI;
- доставлять готовые образы в окружения.

Собственный registry возможен позже, если появится реальная причина:

- нужен полный контроль;
- нужна изоляция;
- нужен собственный инфраструктурный контур.

## Деплой

До появления реального `prod` достаточно одной `dev`-среды.

По текущему направлению:

- `dev` деплоится автоматически;
- `prod` появляется после `MVP`;
- приложения и инфраструктура готовятся так, чтобы поддерживать раздельные окружения без ломки пайплайна.

## Оркестрация

`Docker Swarm` рассматривается как возможный путь для `prod`, но не как обязательный стартовый шаг.

Пока проект развивается одним человеком, разумнее ориентироваться на:

- простой и понятный деплой;
- минимальное количество инфраструктурных сущностей;
- подготовленность к миграции в более сложную схему позже.

Если на старте достаточно одного сервера, можно начинать с более простой модели и не тащить оркестрацию раньше времени.

## Что важно помнить

Главная цель стартовой архитектуры не в том, чтобы сразу сделать “как у большого enterprise-продукта”, а в том, чтобы:

- не заблокировать рост;
- не утонуть в инфраструктуре;
- сохранить ясные границы системы;
- оставить путь к платформенной модели и интеграциям.
